## Pattern To Copy

- Reuse a registry-generated schema for both admin preview and public runtime form rendering.
- Use actual records for submissions that require status, review, filtering, and audit.
- Use modal/slide-over Actions for menu-triggered forms.

## Pattern To Avoid

- Do not store public form definitions in `PublicFormDefinition` records.
- Do not accept arbitrary validation rule strings or PHP class names from JSON.
- Do not add file upload support in v1 unless explicitly approved and tested.

## PodText Adaptation Notes

Form definitions belong in settings JSON. Actual submissions are user-generated records and are the one justified settings exception.

## JSON-First Settings Recommendation

Store `public_forms.definitions` with:

- `key`
- `name`
- `heading`
- `description`
- `submit_label`
- `success_message`
- `display_mode_default`
- `enabled`
- `fields`
- `settings`

Field blocks: text, email, phone, textarea, select, checkbox, toggle, url. File upload is deferred.

## Model/Table Considered

Accepted exception: `PublicFormSubmission`.

Settings JSON is insufficient because submissions are transactional records, can be numerous, need filtering/status/admin review, need timestamps, and should not mutate the definition config.

## Recommended Model/Schema Options

Smallest safe schema:

- `id`
- `form_key`
- `form_name_snapshot`
- `payload` JSON/text
- `status` string enum: `new`, `reviewed`, `archived`
- `submitted_at`
- `source_url`
- `metadata` JSON/text for IP hash/user agent hash/referrer if approved
- timestamps

No relationships required in v1.

## Recommended Filament Patterns

- SettingsPage Builder for form definitions and fields.
- `PublicFormSubmissionResource` only if persistence is implemented.
- Submission table columns: form key, status badge, submitted at day-first, source URL/domain, short payload summary.
- Actions: mark reviewed, archive, restore, delete only if authorized.

## Public Livewire/Blade Implications

Public menu item chooses form key and modal/slide-over display. Runtime form schema is generated by a registry and validates only known field blocks. Disabled forms cannot submit even if a stale menu opens them.

## Tests

- Enabled form submits with valid payload.
- Disabled form cannot submit.
- Unknown field type is ignored/rejected.
- Validation generated from registry works for email/url/required/options.
- Submission payload is escaped in admin.
- Status transitions work.

## Security Notes

No arbitrary rules, classes, HTML, or file uploads in v1. Rate limiting/honeypot is recommended before public launch. Payload values must be escaped and CSV/export formula injection considered if exported later.

## Open Questions
