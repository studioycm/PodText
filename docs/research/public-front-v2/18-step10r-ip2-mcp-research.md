Schema evidence:

- `content_items.published_at` exists and is the "published on site" date.
- `content_items.original_published_at` exists and is the original/external publication date.
- `content_items.external_thumbnail_url` exists and can be used as the episode image.
- `content_groups.cover_path` exists and can be used as the podcast fallback image.
- `transcriptions.published_at` exists and remains the selected/effective transcription date.
- `content_items.duration_seconds`, `transcriptions.word_count`, content item categories, content group categories, and enabled content tags already support the IP2 info-line data.
- `settings.public_content.item_page` already exists from IP1 and can be extended by a settings migration.

Documentation searches:

- `Filament SettingsPage tabs sections repeater nested array state path`
  - Relevant findings: Filament 5 settings pages can bind nested array paths directly; repeaters are appropriate for ordered arrays; repeaters should use finite `Select` controls and `itemLabel()` for manageable JSON lists.
- `Laravel Eloquent constrained eager loading prevent lazy loading`
  - Relevant findings: constrained eager loading should be explicit with `with([... => closure])`; avoid fallback relationship queries under non-production lazy-loading prevention.
- `Pest Laravel rendered output assertSee html false`
  - Relevant findings: rendered-output tests can assert literal HTML with `assertSee(..., false)` and should prove the real public page workflow rather than class existence.

Boost adaptation notes:

- Keep the existing `PublicContentItemQueries::base()` relation loading path. IP2 should not introduce per-render queries in the Blade view.
- Use the existing `PublicFrontRenderContext::itemPage()` accessor from IP1.
- Extend the existing `item_page` JSON settings group instead of adding a second settings group or new model.
- Add one Spatie settings migration for existing settings rows so saved IP1 rows receive IP2 keys.

## FilamentExamples MCP Access

Access level: `search_examples` snippets only. No source/detail/fetch tool was available, so this is search/snippet research rather than full source inspection.

First query batch:

- `public detail page`
- `media sidebar page`
- `SettingsPage tabs`
- `settings repeater`

Findings:

- Account/settings examples showed SettingsPage state bound through a `data` array and composed with tabs/sections.
- Repeater examples showed `collapsed()`, `reorderable()`, finite selects, and `itemLabel()` for ordered nested configuration rows.
- Custom detail page examples favored preparing display data in the page class and keeping Blade mostly presentational.

Second query batch:

- `badge icon label`
- `metadata row renderer`
- `public card layout`
- `custom page view data`

Findings:

- Profile/detail examples prepared avatar/image/stat metadata in `getViewData()` or page methods, then rendered fixed class maps in Blade.
- Badge-like metadata examples used finite styling decisions, not raw class strings in persisted state.
- Public card/layout snippets reinforced centralizing class maps and passing prepared link/view arrays to Blade.

Refined query batch:

- `getViewData public page`
- `nested settings repeater fields`
- `metadata badge links`
- `public Blade card links`

Findings:

- Repeater rows should stay simple and finite: field key, label mode, icon key, icon position, size, color.
- Public page links should be generated by route helpers and escaped in Blade.
- The PodText adaptation should reuse the existing app-owned card part shell and icon resolver for labels/icons instead of a new free-form renderer.
